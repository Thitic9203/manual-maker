## บทเรียนจากรอบรีวิวนี้ (สำคัญ — คงไว้เตือนรอบหน้า)

> **"เขียนเสร็จ" ≠ "หน้าที่เกี่ยวข้องทั้งหมด sync ตาม"** — เมื่อแก้เนื้อหาหน้าลูก (เพิ่ม/ลบ UC) ต้องเช็ก
> "หน้าดัชนี"/หน้าอื่นที่ summarize ตัวเลขจากหน้าลูกนั้นเสมอ ว่าซิงก์ตามหรือไม่ — action 12b
> (เพิ่ม UC-4/UC-5/OLS-14/15 ใหม่) แก้ 4 หน้าลูกสำเร็จและ verify ทุกหน้าจริง แต่**ไม่ได้ย้อนไปแก้ตัวเลข
> สรุปที่หน้าดัชนีที่ผูกกับ 4 หน้านั้น** ทำให้รายงาน "เสร็จ 17/17" ที่ให้ผู้ใช้ไปรอบก่อนนั้น ถูกต้องเฉพาะ

  - Story ที่ไม่มี description ใน Jira → ยังต้องเข้า PRD/UC (ถ้าไม่ backlog) แต่ทำเครื่องหมาย TBC

## ความเสี่ยงที่พบระหว่างงาน (ต้องระวังในรอบต่อไป)

### connector คืนผลผิดแบบสุ่ม — พิสูจน์แล้ว ไม่ใช่การเดา
- `getConfluencePage` ด้วย id เดิม เคยคืน **หน้าคนละหน้า** และครั้งหนึ่งผิดครบ 5/5
- `searchJiraIssuesUsingJql` เคยขอตั๋วใบหนึ่ง แต่ **คืนตั๋วคนละใบ**
- `updateConfluencePage` เคยคืน **response เป็นหน้าอื่น** (แต่เขียนลงถูกหน้า — response เชื่อไม่ได้)
- **อาการผูกกับ `contentFormat`**: `html` คืนถูกราว 1 ใน 3 · `markdown`/`adf` คืนถูกทุกครั้งที่ทดสอบ

**มาตรการที่ใช้ (ต้องคงไว้ทุกครั้งที่แตะ connector):**
1. ทุกการอ่าน → เทียบ id/คีย์ที่คืนกลับกับที่ขอ **ไม่ตรง = ทิ้งทันที ดึงใหม่** (สูงสุด 5–8 ครั้ง)
2. อ่านยืนยันตัวตนด้วย `markdown`/`adf` ก่อน แล้วค่อยใช้ `html` เมื่อจำเป็นต้องได้ body
3. หลัง `update` **ห้ามเชื่อ response** ต้องอ่านกลับด้วย `markdown`/`html` ยืนยันว่าเขียนถูกหน้า+เนื้อหาตรง
4. ครบโควตายังไม่ตรง → บันทึกว่า "ตรวจไม่ได้" **ห้ามเดา**

### Jira query บางแบบ timeout
- `parent = "<EPIC>"` และ `key in (หลายใบ)` — timeout บ่อย โดยเฉพาะ Epic ใหญ่ (พบใน OLS-3: OLS-45, OLS-210)
- `key = <ใบเดียว>` — ตอบเร็ว ใช้ได้จริง → **ดึงทีละใบเสมอ**
- มีบางตั๋วที่ timeout เฉพาะใบ แม้ query เบา (ดู action ข้อ 7)

### Thai text corruption จากการ retype เนื้อหายาวเข้า tool argument — พิสูจน์แล้ว ไม่ใช่การเดา
- retype/reconstruct เนื้อหา HTML ที่มีภาษาไทยยาว (25KB+) จาก memory เข้าไปเป็น tool argument
  ทำให้ตัวอักษรบางจุดเพี้ยนแบบเงียบ (ธ→ท, ถ→ท, ณ→ญ, สระเพี้ยน) **แม้ในส่วนที่ไม่ได้ตั้งใจแก้**
- เกิดขึ้นจริงกับ OLS-2 ตอนเขียน UC-4 ครั้งแรก (รอบ 6)

**มาตรการที่ใช้ (Corruption Avoidance Protocol — ต้องคงไว้ทุกครั้งที่แก้เนื้อหาไทยยาว):**
1. ทำการแก้ไขทุกครั้งด้วย Python string operation บนไฟล์ local เท่านั้น (ห้าม retype จาก memory)
2. อ่านไฟล์ HTML ที่เตรียมสุดท้ายด้วย Read tool ก่อน push ทุกครั้ง เพื่อยืนยันเนื้อหาตรงก่อนยิงจริง
3. ใช้ค่า string ตรงจากไฟล์เป็น body argument ของ `updateConfluencePage` — ไม่พิมพ์ซ้ำจากความจำ
4. push แล้ว **ต้อง**อ่านกลับด้วย `html`/`markdown` แล้ว diff plain-text เทียบไฟล์ local ให้ตรง 1:1
5. ถ้าการแก้เป็น ASCII เท่านั้น (เลข/คำอังกฤษ) ความเสี่ยง corruption ต่ำกว่ามาก แต่ยังต้องยืนยันตามข้อ 2 และ 4 เสมอ
