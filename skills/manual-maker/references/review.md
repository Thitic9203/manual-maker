# รีวิว 5 ชั้น + Delivery Gate (บังคับ — ห้ามข้าม)

คู่มือที่ผ่านการ draft แล้ว **ยังไม่ใช่ของที่ส่งได้**. ก่อนบอกผู้ใช้ว่าเสร็จ ต้องผ่าน **ครบทั้ง 5 ชั้น**
โดยแต่ละชั้นมี **หลักฐาน** ไม่ใช่ความรู้สึก.

## กฎของ Gate (ข้อเดียวที่สำคัญที่สุด)

> **ห้ามบอกว่า "เสร็จแล้ว" / "ส่งงานได้" / "เรียบร้อย" จนกว่าทั้ง 5 ชั้นจะขึ้น ✅ ครบ.**

ผลที่ตามมา — บังคับทั้งหมด:

- **ตรวจไม่ได้ = ไม่ผ่าน.** ไม่มีสถานะ "น่าจะผ่าน" / "ไม่น่ามีปัญหา" / "ข้ามไปก่อน". ถ้าตรวจชั้นไหนไม่ได้
  (ไฟล์ยังไม่ export, ไม่มีต้นแบบมาเทียบ, ไม่รู้ว่าผู้ใช้สั่งวงแดงหรือไม่) → ชั้นนั้น **ไม่ผ่าน** แล้ว **ถามผู้ใช้**
  ห้ามเดาแทน.
- **เจอ FAIL ข้อเดียว = แก้ แล้วรีวิว _ใหม่ทั้ง 5 ชั้น_.** ไม่ใช่รีวิวเฉพาะข้อที่แก้ — การแก้ชั้นหนึ่งพังอีกชั้นได้บ่อย
  (แทรกรูปเพิ่ม → เลขข้อเลื่อน; แก้คำศัพท์ → TOC ไม่ตรง).
- **ตรวจจาก _ไฟล์ที่ export จริง_ เท่านั้น.** ห้ามรีวิวจาก draft ที่อยู่ในบทสนทนา หรือจากความจำว่าเขียนอะไรไป —
  ข้อผิดพลาดส่วนใหญ่เกิด**ตอนแปลงเป็นไฟล์** (ฟอนต์เพี้ยน รูปหลุด TOC ไม่อัปเดต คำพราก).
- **ห้ามใช้คำคลุมเครือในผลรีวิว** — "น่าจะ", "คาดว่า", "ควรจะ", "โดยทั่วไป". เขียนสิ่งที่ **เห็นจริง** เท่านั้น.
- **ห้ามส่งพร้อมข้อแม้.** ห้ามพูดว่า "ส่งให้ก่อน เดี๋ยวแก้ทีหลัง" หรือ "เหลือแค่เรื่องเล็กน้อย" — ยังไม่ผ่าน = ยังไม่ส่ง.

## แหล่งอ้างอิงของการรีวิว (Source of truth)

ทุกชั้นตัดสินโดยเทียบกับ **ตารางยืนยันจาก Step 2** (intake ที่ผู้ใช้ confirm) เท่านั้น —
ไม่ใช่ความเห็นของผู้เขียน ไม่ใช่ "แบบที่คู่มือทั่วไปเขาทำกัน".
ถ้าตารางยืนยันไม่ได้ระบุเรื่องไหนไว้ และเรื่องนั้นจำเป็นต่อการตัดสิน → **ถามผู้ใช้ อย่าเดา**.

## รีวิวทีละหัวข้อย่อยระหว่างทาง ≠ gate นี้

ระหว่าง Step 4–6 มี `manual-section-reviewer` รีวิวทุกหัวข้อย่อยทันทีที่เขียนเสร็จ (ดู `parallel.md`)
เพื่อจับข้อผิดตั้งแต่ยังแก้ถูก **แต่มันไม่ได้แทนที่รีวิว 5 ชั้นนี้ และไม่ลดข้อไหนของที่นี่เลย**:

- ผู้รีวิวรายหัวข้อตัดสินได้แค่ **ชั้น 1–4 ในระดับหัวข้อ** จากดราฟต์กับไฟล์ภาพ
- ข้อผิดที่เหลือ **เกิดตอนแปลงเป็นไฟล์** จึงพิสูจน์ได้เฉพาะบนไฟล์ที่ export แล้ว — คำพราก · ฟอนต์
  fallback · รูปหลุด · TOC ไม่ตรง · เลขข้ามบท · **ชั้นที่ 5 ทั้งชั้น**
- "ทุกหัวข้อผ่านรีวิวรายหัวข้อแล้ว" **ไม่ใช่หลักฐาน**ของชั้นใดในตารางนี้ ห้ามยกมาอ้างแทนการตรวจจริง

ผลของลูปนั้นคือ gate นี้มักผ่านตั้งแต่รอบแรก ไม่ใช่ gate นี้สั้นลง.

## ชั้นที่ 1 — ตรงตามที่ผู้ใช้ยืนยัน (Requirement conformance)

**มุมมอง:** ของที่ส่ง ตรงกับที่ผู้ใช้สั่งไว้ตอนต้นหรือไม่ — ทีละบรรทัดของตารางยืนยัน.

ตรวจ:
- ขอบเขต/โมดูล ครบตามที่ยืนยัน **และไม่เกิน** (ห้ามแถมบทที่ไม่ได้สั่ง)
- **การแบ่งเล่ม (Q9)** — จำนวนเล่มและมิติการแบ่งตรงตามที่ยืนยัน; เล่มแยกต้องมีปก/TOC/เลขข้อของตัวเอง เริ่มที่ 1
- ภาษา, ฟอนต์+ขนาด, output format, ปลายทาง (Confluence space/parent) ตรงตามที่ยืนยัน
- **โหมดภาพ ตรงตามที่สั่ง** — ผู้ใช้สั่ง "มีวงแดง" หรือ "ไม่มีวง" → ทำตามนั้น **ทั้งสองแบบถูกได้**
  ผิดคือ *ไม่ตรงกับที่สั่ง* ไม่ใช่ *ไม่มีวง*

**หลักฐานที่ต้องแสดง:** ตารางเทียบ `สิ่งที่ยืนยัน → สิ่งที่ส่งจริง` ทุกแถว.

---

## ชั้นที่ 2 — ทุกอย่างมีที่มา ห้ามมโน (Sourcing / anti-fabrication)

**มุมมอง:** ทุกประโยคสาวกลับไปถึงแหล่งได้ไหม.

ตรวจ:
- ทุกขั้นตอน มาจาก **ระบบจริง + แหล่งที่ผู้ใช้ให้** (Confluence/spec/ต้นแบบ) — ไม่มีขั้นตอนที่แต่งขึ้น
- ชื่อเมนู/ปุ่ม/แท็บ/ฟิลด์ **สะกดตรงกับระบบจริง** ไม่ใช่คำที่คิดเอง
- **บทที่ไม่มีแหล่ง = ไม่มีบทนั้น** — Troubleshooting/FAQ (Q20) และ Support (Q19) ไม่มีแหล่ง → **ตัดทิ้ง**
  ห้ามแต่งปัญหา วิธีแก้ เบอร์ติดต่อ อีเมล หรือเวลาทำการขึ้นมาเอง
- ตัวเลข เงื่อนไข สิทธิ์การเข้าถึง ไม่ได้เดาเอา

**หลักฐานที่ต้องแสดง:** รายการ `หัวข้อ → แหล่งที่มา` ครบทุกหัวข้อ. หัวข้อไหนเขียนแหล่งไม่ได้ → **ชั้นนี้ไม่ผ่าน**.

## ชั้นที่ 3 — ภาพ (Figures)

**มุมมอง:** ภาพเป็นของจริง เต็มจอ และวงตรงตามที่สั่ง.

ตรวจ (รายละเอียดผูกกับ `screenshots.md`):
- **ของจริงทุกรูป** — ไม่มีกล่อง placeholder ไม่มี mock-up ไม่มีตารางวาดแทนหน้าจอ
- **เต็มจอ ไม่ crop เนื้อหา** ภาพคมชัด อ่านออก
- **โหมดวงตรงกับที่ผู้ใช้สั่ง** (ดูชั้นที่ 1):
  - สั่งให้มีวง → วงแดงอยู่บนเป้าที่คลิกจริง, **เลขในวงตรงกับเลขขั้นตอน 1:1**, ≤ 5 วง/รูป, สไตล์เดียวกันทั้งเล่ม
  - สั่งไม่ให้มีวง → **ต้องไม่มีวงเลย** ไม่ใช่ "ใส่ไว้เผื่อ"
- ไม่มีเคอร์เซอร์ ไม่มีขอบเรืองแสง โลโก้หน่วยงานไม่ถูกกินหาย
- **ปิดชื่อคนจริง** (ผู้เรียนเป็นเยาวชน)
- รูปฝังในไฟล์จริง — `word/media/` มีไฟล์ และ relationship ไม่หลุด

**ชื่อคนและรหัสนักเรียน — ต้องมีหลักฐาน ไม่ใช่ความรู้สึก.** การลบชื่อทำที่ DOM ก่อนกดชัตเตอร์ และ**ต้อง fail closed** — อ่านหน้าเว็บซ้ำหลัง scrub แล้วโยน error ถ้ายังเจอชื่อ ห้ามถ่ายต่อ. ก่อน build ให้ครอปแถบบัญชีผู้ใช้ของ**ทุกภาพ**มาต่อเป็นแผ่นเดียวแล้วดูด้วยตา — รอบจริงที่ผ่านมา สามรอบแรกรายงานว่าสะอาดทั้งที่ยังมีชื่อครูหลุดอยู่หนึ่งภาพ. **ตรวจไม่ได้ = ไม่ผ่าน** ใช้กับข้อนี้เต็มที่.


### "เลขในวงตรงขั้นตอน 1:1" — ตอนนี้มีหลักฐานบังคับ ไม่ใช่การกวาดตาดูอีกต่อไป

ข้อนี้เคยเป็นแค่ประโยคให้คนดูเอง และ**หลุดมาแล้วจริง**: คู่มือที่ส่งไปมีวง ② อยู่บนปุ่มของขั้นตอนที่ 3,
วง ④ อยู่บนทางเลือกที่ไม่ใช่ขั้นตอนที่มีเลข, และวง ⑤ ถูกวาดซ้ำสองรูป — เพราะวงถูกวางตามดราฟต์ก่อนหน้า
แล้วไม่ได้ไล่ใหม่ตอนเรียบเรียงขั้นตอนใหม่ ไม่มีอะไรในรีวิวจับได้เลย

ตอนนี้ต้องรัน **`scripts/verify-annotations.py`** เทียบ `annotations.json` กับ **พิกเซลจริง** ในไฟล์ PNG:

```bash
MM=$(ls -d ~/.claude/plugins/cache/*/manual-maker/*/skills/manual-maker 2>/dev/null | sort -V | tail -1); [ -n "$MM" ] || MM=~/.claude/skills/manual-maker
/usr/bin/python3 "$MM/scripts/verify-annotations.py" manual-assets/<slug>/ \
    --assets manual-assets/<slug>/ --docx <ไฟล์.docx>
```

จับได้: เลขไม่ครบ 1..N · เลขซ้ำในหัวข้อเดียวกัน (รวมกรณีวาดข้ามสองรูป) · เลขหลุดช่วง · ไฟล์ที่อ้างไม่มีจริง ·
เกิน 5 วง/รูป · **จำนวนวงที่วาดจริงไม่ตรงกับที่แมนิเฟสต์อ้าง** · **แมนิเฟสต์อ้างพิกัดที่ไม่มีวงอยู่จริง** ·
รูปที่อ้างไม่ได้ถูกฝังในเอกสาร · **`label` ของวงไม่ตรงกับคอนโทรลที่ข้อความขั้นตอนนั้นอ้างไว้**
**Exit 1 = ห้ามส่ง**

### เส้นแบ่ง: อะไรเครื่องตัดสินแล้ว อะไรยังเป็นของคน

เส้นนี้ขยับมาแล้วสองรอบ เขียนไว้ให้ชัดเพื่อไม่ให้เผลออ้างเกินหรือตรวจซ้ำโดยไม่จำเป็น

**เครื่องตัดสินแล้ว (มีหลักฐานบังคับ ไม่ต้องใช้สายตา):**
- เลขในวงครบ `1..N` ไม่ขาด ไม่ซ้ำ ไม่หลุดช่วง — เทียบกับจำนวนขั้นตอนของหัวข้อนั้น
- แมนิเฟสต์ไม่ได้โกหก — จำนวนวงและพิกัดตรงกับ **พิกเซลจริง** ในไฟล์ PNG
- **การจับคู่ วง ↔ ขั้นตอน สอดคล้องกันเชิงข้อความ** — `label` ของวงต้องพูดถึงคอนโทรลตัวเดียวกับที่
  `step_text` ของขั้นตอนนั้นอ้างไว้ในเครื่องหมายคำพูด **และ**ถ้า `label` ไปตรงกับคอนโทรลของ
  *ขั้นตอนอื่นในหัวข้อเดียวกัน* ก็ตกทันที — นี่คือข้อที่ปิดเคสจริง (วง ② ไปนั่งบนปุ่มที่ขั้นตอนที่ 3 พูดถึง)

**ยังเป็นวิจารณญาณของคน — สคริปต์ไม่มีทางรู้:**
- **คอนโทรลที่ขั้นตอนอ้างไว้ เป็นตัวที่ควรคลิกจริงหรือเปล่า** ถ้าดราฟต์เขียนว่า `คลิกปุ่ม “ยกเลิก”`
  ทั้งที่ควรเป็น `“บันทึก”` และวงก็ทับปุ่มยกเลิกอย่างซื่อสัตย์ → **ผ่านสคริปต์ทุกข้อ** เพราะทุกอย่าง
  สอดคล้องกันหมด ผิดที่ตัวขั้นตอนเอง ไม่ใช่ที่การจับคู่ (ข้อนี้เกี่ยวกับชั้นที่ 2 ด้วย — ขั้นตอนต้องมีที่มา)
- วงทับ **พิกัดที่ถูกต้องบนหน้าจอ** ไหม — สคริปต์รู้แค่ว่ามีวงอยู่ตรงพิกัดที่อ้าง ไม่รู้ว่าใต้วงคือปุ่มอะไร
- ขั้นตอนที่ **ไม่ได้อ้างคอนโทรลในเครื่องหมายคำพูด** (เช่น `เปิดเบราว์เซอร์ แล้วเข้าที่ https://…`,
  `คลิกคาบเรียนที่ต้องการในตาราง`) — สคริปต์ข้าม ต้องดูเอง วัดจากเนื้อหาจริง: **ประมาณ 4 ใน 22 ขั้นตอน**
- ภาพเป็นหน้าจอที่ถูกต้อง เต็มจอ คมชัด ไม่มีเคอร์เซอร์ ปิดชื่อคนครบ

> **ตรวจไม่ได้ = ไม่ผ่าน** ยังใช้เต็มที่กับทุกข้อในกลุ่มหลัง สคริปต์ผ่าน **ไม่ได้แปลว่าชั้นที่ 3 ผ่าน**

**หลักฐานที่ต้องแสดง:** ผลรัน `verify-annotations.py` (ตารางเต็ม) **บวก** ตารางต่อรูปที่คนตรวจเอง —
`ไฟล์ → หัวข้อ/ขั้นตอน → เลขวงที่มี → label → คอนโทรลที่ขั้นตอนอ้าง เป็นตัวที่ควรคลิกจริงไหม (ยืนยันด้วยตา)`.
แถวที่สคริปต์ขึ้น **ข้าม** ต้องมีบรรทัดยืนยันด้วยตาเสมอ. ผลสคริปต์อย่างเดียว **ไม่พอ** สำหรับชั้นนี้.

## ชั้นที่ 4 — ตัวหนังสือและตัวเลข (Text integrity)

**มุมมอง:** อ่านแล้วไม่สะดุด ไม่มีคำผิด ไม่มีคำพราก เลขไม่เพี้ยน.

ตรวจ:
- **เลขข้อถูกทั้งหมด** — ต่อเนื่อง ไม่ข้าม ไม่ซ้ำ; **TOC ตรงกับเนื้อใน**; เลขหน้าใน TOC ตรงของจริง
- **ไม่มีคำผิด** สะกดถูก ใช้คำเป็นทางการ
- **ไม่มีคำพราก** — ห้าม "นัก" ท้ายบรรทัด แล้ว "เรียน" ขึ้นบรรทัดใหม่. ต้นเหตุมี 2 อย่าง และตรวจได้จากไฟล์:
  1. มี **ช่องว่าง/บรรทัดใหม่แทรกกลางคำ** ในต้นฉบับ
  2. run ภาษาไทย **ไม่ได้ตั้ง `w:lang w:bidi`** → Word ไม่มีพจนานุกรมไว้ตัดคำ เลยตัดกลางคำ
     (และ `w:cs` หายก็ทำให้ฟอนต์ fallback)
- **คำศัพท์ล็อกเดียวทั้งเล่ม** — เลือก "ผู้เรียน" แล้วห้ามมี "นักเรียน"/"นร."/"ผู้ใช้" ปนเข้ามา
- **โทนภาษา** — ทางการ ไม่มีสรรพนามบุรุษ 1/2 (ผม/ฉัน/คุณ/ท่าน) ไม่มีคำลงท้าย (ครับ/ค่ะ/นะ) ไม่ใช่ภาษาแปล
- **ไม่มี placeholder ค้าง** — `TODO`, `[ระบุ…]`, `SCREENSHOT PLACEHOLDER`

**หลักฐานที่ต้องแสดง:** ผลรัน `scripts/verify-doc.py` + จุดที่ตรวจด้วยตาแล้ว (โทน/คำผิด ที่เครื่องตัดสินแทนไม่ได้).

## ชั้นที่ 5 — รูปเล่ม (Document format)

**มุมมอง:** หน้าตาเอกสารตรงฟอร์แมตที่ผู้ใช้กำหนด.

ตรวจ:
- **หน้าปก** ตามที่ผู้ใช้กำหนด — ชื่อระบบ เวอร์ชัน วันที่ กลุ่มผู้อ่าน ครบตามฟอร์แมต
- **header** ทุกหน้าตามต้นแบบ (โลโก้ + บรรทัดหน่วยงาน)
- **footer + เลขหน้า** — ต้องเป็น `PAGE` field จริง ไม่ใช่เลขที่พิมพ์มือ
- **TOC เป็น field จริง** (อัปเดตได้) ไม่ใช่ตารางพิมพ์แข็ง
- **ฟอนต์และขนาดสม่ำเสมอทั้งเล่ม** ตามที่ยืนยัน; ภาษาไทยตั้ง `w:rFonts` ครบ **ทั้ง 4 ช่อง** รวม `w:cs`
- **ถ้ามีต้นแบบ (ต้นฉบับ)** — ปก/header/footer/TOC/styles ต้องเป็น**ของต้นแบบจริง** ไม่ใช่ของที่สร้างเลียนแบบ
- ส่งมอบ Word ถูกวิธี — ถ้ามีไฟล์ล็อก `~$….docx` บอกผู้ใช้ให้ปิด Word **โดยไม่ Save** เปิดใหม่ แล้วตอบ
  **"Update fields? → Yes"** เพื่อให้ TOC/เลขหน้ารีเฟรช

**หลักฐานที่ต้องแสดง:** ผลรัน `scripts/verify-doc.py` ข้อ 6 + การเทียบกับต้นแบบทีละองค์ประกอบ.

## ตัวช่วยเชิงกล — `scripts/verify-doc.py` + `scripts/verify-annotations.py`

สองตัวนี้แบ่งกันคนละด้าน และ **ต้องรันทั้งคู่** (ตัวหลังข้ามได้เฉพาะเมื่อโหมดภาพเป็น `none`):

| สคริปต์ | ตรวจอะไร | ชั้นที่ให้หลักฐาน |
|---|---|---|
| `verify-doc.py` | ตัวเอกสาร — ฟอนต์ ตัดคำ TOC รูปฝัง placeholder credential | 4, 5 |
| `verify-annotations.py` | วงแดง — เลขครบ/ไม่ซ้ำ/ไม่หลุดช่วง · แมนิเฟสต์ตรงกับพิกเซลจริง · `label` ตรงกับคอนโทรลใน `step_text` | 3 |

รันกับ **ไฟล์ .docx ที่ build เสร็จแล้ว**:

สคริปต์อยู่**ข้าง `SKILL.md`** ไม่ใช่ในโปรเจกต์ของผู้ใช้ — ต้อง resolve path ก่อนเสมอ
(ดู *Running this skill's scripts* ใน `SKILL.md`) ไม่งั้นจะหาไม่เจอ:

```bash
MM=$(ls -d ~/.claude/plugins/cache/*/manual-maker/*/skills/manual-maker 2>/dev/null | sort -V | tail -1); [ -n "$MM" ] || MM=~/.claude/skills/manual-maker
/usr/bin/python3 "$MM/scripts/verify-doc.py" <ไฟล์.docx> \
    --terms "ผู้เรียน,ครูผู้สอน,ผู้ดูแลระบบ" \
    --annotations required     # หรือ none ตามที่ผู้ใช้สั่ง
```

ครอบคลุม: placeholder ค้าง · `w:cs` · `w:lang w:bidi` (ต้นเหตุคำพราก) · คำล็อกถูกเว้นวรรคกลางคำ ·
อักขระล่องหน · รูปฝังจริง/rel ไม่หลุด · ปก-header-footer-PAGE-TOC · เลขหัวข้อต่อเนื่อง · credential หลุด.
**Exit 1 = ห้ามส่ง.**

แล้วรันตัววงแดงต่อ (รายละเอียดและขอบเขตอยู่ในชั้นที่ 3 ด้านบน):

```bash
/usr/bin/python3 "$MM/scripts/verify-annotations.py" manual-assets/<slug>/ \
    --assets manual-assets/<slug>/ --docx <ไฟล์.docx>
```

> **สิ่งที่สคริปต์เหล่านี้ตัดสิน_ไม่ได้_ และยังต้องตรวจเอง:** เนื้อหาถูกต้องตามระบบจริงไหม (ชั้น 2),
> ภาพเป็นหน้าจอที่ถูกต้อง**และวงทับปุ่มที่ถูกต้อง**ไหม (ชั้น 3), โทนภาษาและคำผิดเชิงความหมาย (ชั้น 4),
> หน้าตาตรงต้นแบบไหม (ชั้น 5).
> **ผ่านสคริปต์ ≠ ผ่านรีวิว** — สคริปต์พิสูจน์แค่ว่า "ไม่เจอข้อผิดที่ตรวจอัตโนมัติได้" เท่านั้น.

การเช็กแบบละเอียดรายข้อ ใช้ **Final Review Checklist** ใน `template.md` ประกอบ — เนื้อหาเดียวกัน คนละมุมมอง:
checklist คือรายการข้อ, 5 ชั้นคือลำดับการตัดสินและ gate.

## รูปแบบรายงานผล (ต้องพิมพ์ให้ผู้ใช้เห็นก่อนส่งมอบเสมอ)

```
รีวิว 5 ชั้น — <ชื่อเอกสาร> (รอบที่ N)

| ชั้น | มุมมอง | ผล | หลักฐาน |
|---|---|---|---|
| 1 | ตรงตามที่ยืนยัน | ✅ / ❌ | <เทียบกับตารางยืนยัน> |
| 2 | ทุกอย่างมีที่มา | ✅ / ❌ | <หัวข้อ → แหล่ง> |
| 3 | ภาพ | ✅ / ❌ | <ผล verify-annotations.py + ตารางต่อรูปที่ยืนยันด้วยตาว่าวงทับเป้าถูก> |
| 4 | ตัวหนังสือและตัวเลข | ✅ / ❌ | <ผล verify-doc.py + ที่ตรวจด้วยตา> |
| 5 | รูปเล่ม | ✅ / ❌ | <ปก/header/footer/TOC/ฟอนต์> |

สรุป: ผ่าน 5/5 → ส่งมอบได้   |   ไม่ผ่าน <n> ชั้น → แก้แล้วรีวิวใหม่ทั้ง 5 ชั้น (รอบที่ N+1)
```

ถ้ายังไม่ครบ 5/5 → **รายงานว่าไม่ผ่าน บอกว่าจะแก้อะไร แล้วแก้** ห้ามส่งมอบ ห้ามถามผู้ใช้ว่า "เอาไปใช้เลยไหม".
