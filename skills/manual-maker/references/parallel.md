# ทำงานขนานต่อหัวข้อ + รีวิวทีละหัวข้อย่อย (Step 4–6)

คู่มือหนึ่งเล่มมีหลายหัวข้อย่อยที่ **ไม่ขึ้นต่อกัน** — ถ่ายภาพและเขียนขั้นตอนของ 5.2 ไม่ต้องรอ 5.1
เสร็จ ส่วนนี้อธิบายวิธีกระจายงานนั้นให้ **เร็วขึ้นโดยไม่เสียความถูกต้อง** และวิธีรีวิวทีละหัวข้อย่อย
แทนการรีวิวทีเดียวตอนท้าย

**หน่วยของงาน = หัวข้อย่อย** (เช่น 5.2) ทั้ง **ถ่ายภาพ → ใส่วง → เขียนขั้นตอน** ของหัวข้อย่อยนั้น
อยู่กับผู้เขียนคนเดียว ห้ามแยกเป็น "คนถ่ายรูป" กับ "คนเขียน" เพราะกฎ **เลขในวงแดง = เลขขั้นตอน 1:1**
เป็นข้อผูกภายในหัวข้อย่อย แยกคนเมื่อไรกฎนี้พังทันที

## เมื่อไรถึงกระจายงาน

| จำนวนหัวข้อย่อยที่ต้องทำ | ทำอย่างไร |
|---|---|
| 1 | ไม่ต้อง fan-out — main thread ทำเองเหมือนเดิม |
| 2 | ผู้เขียน 2 ตัว |
| ≥ 3 | **ผู้เขียน 3 ตัว** (ค่าเริ่มต้น) |

ผู้รีวิว **1 ตัวเสมอ** ไม่ว่าผู้เขียนกี่ตัว — ผู้รีวิวหลายตัวจะตัดสินคำศัพท์และโทนไม่ตรงกัน
ซึ่งเป็นสิ่งเดียวกับที่รีวิวมีไว้จับ

## ต้องพร้อมครบก่อนกระจายงาน (ห้าม fan-out ก่อนครบ)

ผู้เขียนขนานกันได้ก็เพราะไม่มีอะไรต้องตกลงกันระหว่างทาง สิ่งที่ต้องนิ่งแล้วก่อนส่งงานออก:

1. **ผู้ใช้ยืนยันแล้ว (Step 2)** — ยังไม่ผ่าน gate ห้ามกระจายงาน
2. **โครงหัวข้อและเลขถูกล็อกแล้ว** — main thread แจกเลขหัวข้อย่อยจากตารางยืนยัน ผู้เขียนไม่ตั้งเลขเอง
   เลขจึงไม่มีทางชนหรือข้าม
3. **แหล่งข้อมูลถูกอ่านแล้ว (Step 3)** — ต้นแบบ/Confluence/spec สรุปเป็นข้อความส่งให้ผู้เขียน
   ผู้เขียนไม่ต้องไปไล่อ่านเอง (ไล่อ่านเองซ้ำ 3 ตัว = ช้าและตีความต่างกัน)
4. **รายการคำล็อกยืนยันแล้ว**
5. **`storageState` ถูกเซฟแล้ว** — main thread ล็อกอิน **ครั้งเดียว** ตาม `screenshots.md`
   (headless, อ่านรหัสจาก env) แล้วเซฟเซสชันไว้ ผู้เขียนทุกตัว **อ่าน** เซสชันนั้นใช้ต่อ
   **ห้ามให้ผู้เขียนล็อกอินเอง** — ล็อกอินพร้อมกัน 3 ตัวเสี่ยงบัญชีโดนล็อก และทำให้รหัสผ่านกระจายไปหลายที่โดยไม่จำเป็น

## ไม่ชนกันเพราะออกแบบไว้ ไม่ใช่เพราะโชคดี

| ของ | เจ้าของ |
|---|---|
| `manual-assets/<slug>/<section>-<step>.png` | ผู้เขียนที่ถือหัวข้อย่อยนั้น **คนเดียว** |
| `manual-assets/<slug>/<section>.annotations.json` | ผู้เขียนที่ถือหัวข้อย่อยนั้น **คนเดียว** |
| `manual-drafts/<slug>/<section>.md` | ผู้เขียนที่ถือหัวข้อย่อยนั้น **คนเดียว** |
| ไฟล์ `.docx` / `.pdf` / หน้า Confluence | **main thread เท่านั้น** |
| เลขหัวข้อ | main thread แจกล่วงหน้า |
| `storageState` | main thread เขียน · ผู้เขียนอ่านอย่างเดียว |
| โปรไฟล์ `~/.manual-maker/profiles/` | main thread เท่านั้น |

ชื่อไฟล์ผูกกับเลขหัวข้อย่อย และหัวข้อย่อยถูกแจกแบบไม่ทับกัน → **ไม่มีไฟล์ไหนถูกเขียนพร้อมกันสองที่**
ผู้รีวิวอ่านอย่างเดียว จึงไม่ชนกับใครเลย

**แมนิเฟสต์วงแดงก็แยกไฟล์ด้วยเหตุผลเดียวกัน** — ผู้เขียนแต่ละคนเขียน
`manual-assets/<slug>/<section>.annotations.json` ของตัวเอง **ห้ามมีไฟล์รวมที่ทุกคนเขียนพร้อมกัน**
`scripts/verify-annotations.py` รับ **โฟลเดอร์** แล้วรวมไฟล์เหล่านั้นให้เอง และหยุดด้วย exit 2 ถ้าเจอ
สองไฟล์ประกาศหัวข้อเดียวกัน (แปลว่าแจกงานทับกัน ซึ่งเป็นบั๊กของการแบ่งงาน ไม่ใช่ของเนื้อหา)

แบ่งงานโดยพยายามให้หัวข้อย่อยในบทเดียวกันอยู่กับผู้เขียนคนเดียว (เปิดหน้าจอซ้ำน้อยลง) แต่กติกาที่ห้ามผิดคือ
**ไม่ทับกัน**

## สั่งงาน

ส่ง `Task` ของผู้เขียนทุกตัว **ในข้อความเดียว** ไม่งั้นมันรันทีละตัวและไม่ได้อะไรเลย
subagent ชื่อ **`manual-maker:manual-section-writer`** และ **`manual-maker:manual-section-reviewer`**
ถ้าชื่อไม่ resolve (ติดตั้งเป็น personal skill ไม่ใช่ plugin) ให้ใช้ general-purpose agent แล้ว
**แปะเนื้อหาไฟล์ `agents/*.md` ตัวนั้นเข้าไปใน prompt ทั้งไฟล์** — ห้ามสั่งงานลอย ๆ โดยไม่มีกฎเหล่านั้น

brief ที่ผู้เขียนต้องได้รับทุกตัว:

```
หัวข้อย่อยที่รับผิดชอบ: <เลข + ชื่อ> (ทั้งหมดที่ได้รับ ห้ามแตะนอกนี้)
slug: <slug>
ระบบ + URL ของแต่ละหน้าจอที่ต้องถ่าย
storageState: <path>            # ล็อกอินมาแล้ว ห้ามล็อกอินเอง
โหมดภาพ: required | none        # ตามตารางยืนยัน
รายการคำล็อก: <คำ, คำ, …>
ภาษา + โทน: <จากตารางยืนยัน>
แหล่งข้อมูลของหัวข้อนี้: <ข้อความ/ลิงก์ที่ ingest มาแล้ว ระบุตำแหน่ง>
path เต็มของ: template.md, screenshots.md, scripts/verify-annotations.py
```

ถ้าโหมดภาพเป็น `required` ผู้เขียนต้องคืน path ของ `<section>.annotations.json` มาด้วย —
หัวข้อที่ไม่มีแมนิเฟสต์ = **ยังไม่เสร็จ** เพราะ Step 8 จะพิสูจน์เลขในวงจากไฟล์นั้น

ผู้เขียนที่ติดปัญหาจะคืน `BLOCKED` + คำถาม → **ถามผู้ใช้ในแชท** → ส่งคำตอบกลับไปให้ agent ตัวเดิม
ทำต่อ (`SendMessage`) ไม่ต้องเริ่มหัวข้อนั้นใหม่

## ลูปรีวิวทีละหัวข้อย่อย

ผู้เขียนคืนงานหัวข้อย่อยหนึ่ง → **รีวิวหัวข้อนั้นทันที** ไม่รอตัวอื่น (ตัวอื่นยังทำงานอยู่ ไม่ต้องหยุด):

1. ส่ง `manual-maker:manual-section-reviewer` พร้อม: ตารางยืนยัน, รายการคำล็อก, path ดราฟต์+ภาพ,
   โหมดภาพ, และผลลัพธ์ที่ผู้เขียนคืนมา
2. ผู้รีวิวคืน `FIX` (มีคำตอบเดียวอยู่แล้ว) และ `ASK` (ต้องใช้ดุลพินิจ)
3. **`FIX`** → ส่งกลับให้ผู้เขียนเจ้าของหัวข้อนั้นแก้ ไม่ต้องรบกวนผู้ใช้ (ผู้รีวิวไม่แก้เอง — จะชนกับผู้เขียน)
4. **`ASK`** → **ถามผู้ใช้ในแชทก่อนเสมอ ห้ามตัดสินใจแทน** รวบคำถามของหัวข้อนั้นถามเป็นชุดเดียว
   ได้คำตอบแล้วส่งกลับให้ผู้เขียนแก้
5. แก้เสร็จ → **รีวิวหัวข้อนั้นใหม่ทั้งหัวข้อ** ไม่ใช่เฉพาะจุดที่แก้
6. หัวข้อย่อยที่ยังมี `ASK` ค้าง = **ยังไม่ผ่าน** ห้ามนับเป็นเสร็จ ห้ามเอาไปประกอบไฟล์

## รายงานความคืบหน้าให้ผู้ใช้ — อย่างน้อยทุก 10 นาที

ช่วง Step 4–6 ผู้ใช้มองไม่เห็นว่าเกิดอะไรขึ้น (งานอยู่ใน subagent) จึงต้องรายงานเอง **ทุกครั้งที่หัวข้อย่อย
เปลี่ยนสถานะ และอย่างน้อยทุก 10 นาที**:

```
~40% เสร็จ

| ขั้น | สถานะ |
|---|---|
| Intake / ยืนยัน / อ่านแหล่ง | ✅ เสร็จ |
| จับภาพ | 🟡 17 ไฟล์ — เล่มครูครบ, เล่มผู้เรียนเหลือ 3 หัวข้อ |
| ปิดชื่อ + วงแดงมีเลข | ⬜ ยังไม่เริ่ม |
| ร่างเนื้อหา | ⬜ |
| ประกอบ .docx 2 เล่ม | ⬜ |
| ตรวจ + ส่งมอบ | ⬜ |

คำถามค้างจากรีวิว: 2 ข้อ (ดูด้านล่าง)
```

แถวคือ **ขั้นตอนจริงของรอบนี้** ไม่ใช่รายการตายตัว สถานะ 🟡 ต้องมีตัวเลขที่วัดได้ (กี่ไฟล์ กี่หัวข้อย่อย
ผ่านรีวิวแล้วกี่หัวข้อ) ไม่ใช่คำว่า "กำลังทำ" เฉย ๆ

**ห้ามนับหัวข้อที่ยังมี `ASK` ค้างว่าเสร็จ** และห้ามปัดเปอร์เซ็นต์ให้ดูดีกว่าจริง มีคำถามค้าง = ยกขึ้นมาถามในข้อความนั้นเลย

ตัวช่วยจับเวลา: `hooks/progress-tick.sh` (PostToolUse) เตือนให้พิมพ์ตารางเมื่อเงียบเกิน 10 นาที
ทำงานเฉพาะตอนมีไฟล์ `~/.manual-maker/state/run.active` (สร้างที่ Step 2 ลบที่ Step 9)
**Claude Code ไม่มี hook แบบตั้งเวลา** — ตัวนี้เกาะ event ของ tool call แล้วคุมด้วยนาฬิกา จึงเป็น
*อย่างช้าไม่เกิน ~10 นาทีเมื่อยังมี tool call เดินอยู่* ไม่ใช่เป๊ะทุก 10 นาที การรายงานตามจังหวะที่หัวข้อย่อย
เปลี่ยนสถานะยังเป็นหน้าที่หลัก hook เป็นแค่ตาข่ายกันเงียบ

## สิ่งที่ยังอยู่ใน main thread เสมอ — ห้ามกระจาย

| งาน | ทำไมกระจายไม่ได้ |
|---|---|
| Step 1–2 intake + gate | ต้องคุยกับผู้ใช้ subagent คุยไม่ได้ |
| Step 3 อ่านแหล่งข้อมูล + ล็อกอิน | ผลลัพธ์เป็น input ร่วมของผู้เขียนทุกตัว อ่านซ้ำ 3 ตัวจะตีความต่างกัน |
| Step 5 เรียก `doc-coauthoring` | ใช้กำหนดโครง/เสียง/รูปประโยคของ **ทั้งเล่ม** ครั้งเดียว แล้วส่งเป็นสัญญาให้ผู้เขียนทุกตัวเขียนตาม เรียกแยก 3 ตัวจะได้สำนวนคนละแบบ ซึ่งตรงข้ามกับที่ต้องการ |
| Step 7 ประกอบไฟล์ | ไฟล์เดียว เขียนพร้อมกันไม่ได้ และเลขหน้า/TOC ต้องมองทั้งเล่ม |
| Step 8 รีวิว 5 ชั้น | เป็น gate ต้องเห็นทั้งเล่มพร้อมกัน |
| Step 9 publish | outward-facing ต้องมีคนยืนยันปลายทาง |
| ถามผู้ใช้ทุกกรณี | subagent ไม่มีช่องทาง |

## ความสัมพันธ์กับรีวิว 5 ชั้น — สำคัญ ห้ามเข้าใจผิด

> **รีวิวทีละหัวข้อย่อย ไม่ได้แทนรีวิว 5 ชั้น** และไม่ได้ลดทอนอะไรใน `review.md` เลย

ผู้รีวิวรายหัวข้อตัดสินได้แค่ชั้น 1–4 **ในระดับหัวข้อ** ข้อผิดที่เหลือ **เกิดตอนแปลงเป็นไฟล์**
จึงเห็นได้เฉพาะบนไฟล์จริง: คำพราก · ฟอนต์ fallback · รูปหลุด · TOC ไม่ตรง · เลขข้ามบท ·
ปก/header/footer (ชั้น 5)

ดังนั้น **Step 8 ยังรันครบ 5 ชั้นบนไฟล์ที่ export แล้วเหมือนเดิมทุกประการ** สิ่งที่ลูปนี้ทำคือทำให้
Step 8 ผ่านตั้งแต่รอบแรกบ่อยขึ้น เพราะข้อผิดระดับเนื้อหาถูกจับไปแล้วตอนยังแก้ถูก
